## 1. 系统概述

### 1.1 系统定位

AI Agent Hub 是一个**企业内部的 AI Agent 管理平台**，核心解决三个问题：

1. **降低 AI 使用门槛** — 业务人员通过表单配置即可创建 Agent，不需要写代码
2. **统一管控 AI 能力** — IT 部门统一管理模型接入、工具权限、Token 消耗
3. **保障安全合规** — 敏感操作走审批，所有行为有审计日志

### 1.2 核心设计原则

| 原则 | 说明 |
|------|------|
| **双引擎分离** | Java 做管理面（权限/审批/编排），Python 做执行面（Agent 运行时） |
| **审批即安全** | 所有高风险操作必须经过审批流程，不是事后审计而是事前阻断 |
| **平台化设计** | 不做单个 Agent，做能管理 100 个 Agent 的平台 |
| **企业级优先** | 多租户、RBAC、SSO、审计日志从第一天开始设计 |
| **工具即产品** | 价值不在 Agent 本身，在工具生态 — 越多的企业工具接入，平台粘性越强 |

### 2.2 关键数据流

#### 流程 A：业务人员创建并发布 Agent

```
用户(前端) → REST API → Java引擎 → 写入DB(agent_definition表)
                                    → 通知Python引擎(新Agent注册)
                                    → Python引擎加载Agent配置到内存
                                    → 返回发布结果
```

#### 流程 B：用户与 Agent 对话

```
用户消息(企业微信) → API Gateway → Java引擎(认证/鉴权/限流)
                                    → gRPC → Python引擎
                                              → LLM推理
                                              → 需要调用工具?
                                                 ├─ 无风险: 直接执行 → 返回结果
                                                 └─ 有风险: gRPC → Java引擎审批流
                                                               → 审批通过? 
                                                                  ├─ 是: 继续执行
                                                                  └─ 否: 返回拒绝信息
                                              → 流式返回给用户
                                    → 写入审计日志(audit_log表)
```

#### 流程 C：敏感工具的审批卡点（核心差异化）

```
Python引擎准备调用「保单查询API」
  → 检查工具风险等级(从Java引擎缓存读取)
  → 发现: 风险等级=高, 需要双人审批
  → gRPC → Java引擎: 发起审批请求(agent_id, tool_id, user_id, 上下文)
  → Java引擎: 创建审批工单, 推送通知给审批人
  → 审批人1 通过 ✓
  → 审批人2 通过 ✓
  → Java引擎 → gRPC → Python引擎: 审批通过, 可以执行
  → Python引擎执行工具调用
  → 结果返回给用户
  → 写入审计日志
```

```protobuf
// protos/agent_hub.proto

service AgentExecutionService {
    // Java → Python: 发起 Agent 执行（双向流）
    rpc ExecuteAgent(stream AgentRequest) returns (stream AgentResponse);
    
    // Java → Python: 停止正在执行的 Agent
    rpc StopAgent(StopRequest) returns (StopResponse);
    
    // Python → Java: Agent 执行过程中发起审批请求
    rpc RequestApproval(ApprovalRequest) returns (ApprovalResponse);
}

service ToolRegistryService {
    // 工具注册（Python 端工具列表同步到 Java 端）
    rpc RegisterTool(ToolDefinition) returns (RegisterResponse);
    
    // Java → Python: 健康检查
    rpc HealthCheck(Empty) returns (HealthStatus);
}

message AgentRequest {
    string session_id = 1;
    string agent_id = 2;
    string user_id = 3;
    string message = 4;
    map<string, string> context = 5;  // 上下文变量
}

message AgentResponse {
    enum ResponseType {
        TEXT_CHUNK = 0;      // 流式文本片段
        TOOL_CALL = 1;       // 工具调用请求
        TOOL_RESULT = 2;     // 工具调用结果
        APPROVAL_NEEDED = 3; // 需要审批
        ERROR = 4;           // 错误
        DONE = 5;            // 完成
    }
    ResponseType type = 1;
    string content = 2;
    map<string, string> metadata = 3;
}

message ApprovalRequest {
    string session_id = 1;
    string agent_id = 2;
    string tool_id = 3;
    string user_id = 4;
    string reason = 5;       // 为什么需要审批
    string context = 6;      // 审批上下文（JSON）
}
```

### 4.3 Agent 执行引擎核心逻辑

```python
# agent_runtime/core/engine.py 的核心执行流程（伪代码）

class AgentEngine:
    """
    Agent 执行引擎 — 这是整个 Python 运行时的核心
    """
    
    async def execute(self, request: AgentRequest) -> AsyncGenerator[AgentResponse]:
        """
        执行 Agent，这是一个异步生成器，流式返回结果
        """
        session = await self.session_manager.get_or_create(request.session_id)
        
        # 1. 构建消息上下文
        messages = session.get_messages()
        messages.append(HumanMessage(content=request.message))
        
        # 2. 加载 Agent 配置
        agent_config = await self.load_agent_config(request.agent_id)
        llm = self.llm_client.get_model(agent_config.model)
        
        # 3. 绑定工具（已通过权限校验的）
        tools = await self.load_authorized_tools(agent_config.tools, request.user_id)
        llm_with_tools = llm.bind_tools(tools)
        
        # 4. 执行 Agent 循环（ReAct 模式）
        while True:
            # 4.1 调用 LLM
            response = await llm_with_tools.ainvoke(messages)
            
            # 4.2 判断是文本输出还是工具调用
            if response.is_text:
                yield AgentResponse(type=TEXT_CHUNK, content=response.text)
            
            elif response.is_tool_call:
                for tool_call in response.tool_calls:
                    # 4.3 检查工具是否需要审批
                    if tool_call.risk_level != "low":
                        approval = await self.grpc_client.request_approval(
                            session_id=request.session_id,
                            tool_id=tool_call.tool_id,
                            reason=f"Agent {request.agent_id} 调用工具 {tool_call.name}",
                            context=tool_call.args
                        )
                        if not approval.granted:
                            yield AgentResponse(type=ERROR, content="操作需要审批，已提交审批请求，请等待审批通过后重试")
                            return
                    
                    # 4.4 执行工具（在沙箱中）
                    result = await self.tool_executor.execute(tool_call, sandbox=True)
                    yield AgentResponse(type=TOOL_RESULT, content=result)
                    messages.append(ToolMessage(content=result))
            
            else:
                # 对话结束
                yield AgentResponse(type=DONE, content="")
                break
            
            # 4.5 更新会话
            await self.session_manager.save(session, messages)
```

### 4.4 自定义工具 SDK 设计

开发者只需要写一个 Python 文件，放到指定目录，平台自动识别：

```python
# 示例: tools/custom/crm_query.py
# 开发者把这个文件放到 python-engine/agent_runtime/tools/custom/ 下即可

from agent_runtime.tools.base import AgentTool, tool

@tool(
    name="客户信息查询",
    description="根据手机号查询客户在CRM中的信息，包括姓名、会员等级、累计保单金额",
    risk_level="medium",          # low / medium / high — 决定审批策略
    rate_limit=100,                # 每天最多调用 100 次
    timeout=10,                    # 超时时间（秒）
    category="CRM"                 # 工具分类，前端展示用
)
class CRMQueryTool(AgentTool):
    """
    参数 Schema 自动从函数签名 + docstring 生成
    """
    
    async def execute(self, phone: str) -> dict:
        """
        Args:
            phone: 客户手机号，11位
            
        Returns:
            dict: 客户信息
        """
        # 这里写实际的业务逻辑
        customer = await crm_client.get_by_phone(phone)
        return {
            "name": customer.name,
            "member_level": customer.level,
            "total_policy_amount": customer.total_amount
        }
```

**效果**: 平台扫描 `tools/custom/` 目录 → 自动解析 `@tool` 装饰器 → 生成 JSON Schema → 同步到 Java 引擎的工具注册中心 → 业务人员在创建 Agent 时就能看到并勾选这个工具了。

## 5. gRPC 通信协议

### 5.1 为什么选 gRPC 而不是消息队列？

| 维度 | gRPC 双向流 | 消息队列 (RabbitMQ/Kafka) |
|------|------------|--------------------------|
| **实时性** | 亚毫秒级 | 数十毫秒 ~ 秒级 |
| **流式返回** | 原生支持 Server Streaming | 需要额外设计 |
| **审批卡点** | 同步等待审批结果 | 异步，实现复杂 |
| **类型安全** | Protobuf 强类型 | 字符串序列化，易出错 |
| **运维复杂度** | 低（无中间件） | 需要额外运维 MQ 集群 |

### 5.2 Proto 完整定义

```protobuf
syntax = "proto3";

package agenthub;

// ==================== Agent 执行服务 ====================

service AgentExecution {
    // 双向流：Java 发送用户消息，Python 流式返回执行结果
    rpc StreamExecute(stream ExecutionRequest) returns (stream ExecutionResponse);
    
    // 单向调用：Java 通知 Python 停止某个会话
    rpc StopSession(StopSessionRequest) returns (StopSessionResponse);
}

// ==================== 工具管理服务 ====================

service ToolRegistry {
    // Python → Java: 注册/更新工具定义
    rpc SyncTool(ToolDef) returns (SyncResult);
    
    // Python → Java: 批量同步工具列表
    rpc SyncTools(stream ToolDef) returns (SyncResult);
    
    // Java → Python: 检查工具是否可用
    rpc CheckToolHealth(ToolHealthRequest) returns (ToolHealthResponse);
}

// ==================== 审批回调服务 ====================

service ApprovalCallback {
    // Java → Python: 审批结果通知
    rpc OnApprovalResult(ApprovalResult) returns (Empty);
}

// ==================== 消息定义 ====================

message ExecutionRequest {
    string session_id = 1;
    string agent_id = 2;
    string tenant_id = 3;
    string user_id = 4;
    string user_name = 5;
    string message = 6;
    string channel = 7;  // wechat / dingtalk / web / api
    map<string, string> variables = 8;  // 上下文变量
}

message ExecutionResponse {
    enum Type {
        THINKING = 0;       // 思考中（可展示 loading 动画）
        TEXT = 1;           // 文本片段（流式）
        TOOL_START = 2;     // 开始调用工具
        TOOL_END = 3;       // 工具调用完成
        APPROVAL_WAIT = 4;  // 等待审批
        ERROR = 5;          // 错误
        COMPLETE = 6;       // 完成
    }
    Type type = 1;
    string content = 2;
    string tool_name = 3;
    string tool_id = 4;
    double token_count = 5;  // 本次消耗 Token 数
    map<string, string> metadata = 6;
}

message ToolDef {
    string tool_id = 1;
    string name = 2;
    string description = 3;
    string category = 4;
    string risk_level = 5;   // low / medium / high
    int32 rate_limit = 6;    // 每天调用上限
    int32 timeout_seconds = 7;
    string json_schema = 8;  // 参数 JSON Schema
    string python_module = 9; // Python 模块路径
}

message ApprovalResult {
    string session_id = 1;
    string approval_id = 2;
    bool granted = 3;
    string comment = 4;
    string approver_id = 5;
}

message Empty {}
```

### 7.2 审批卡点设计（核心差异化）

这是整个产品最关键的差异化能力。设计要点：

| 维度 | 设计 |
|------|------|
| **触发时机** | Agent 运行时在执行工具调用**之前**，实时检查 |
| **风险定级** | 由管理员在工具注册时设定，不可在 Agent 创建时降级 |
| **审批策略** | 低风险=无需审批 / 中风险=单人审批 / 高风险=双人审批 |
| **超时处理** | 超过 N 分钟未审批，自动拒绝 + 通知申请人 |
| **审批代理** | 审批人不在时可设置代理人 |
| **紧急通道** | 特定角色有"紧急绕过"权限，但会强制记录特殊审计日志 |

```
审批决策树：

工具调用请求
  │
  ├─ 风险等级 = low
  │   └─→ 🟢 直接执行（无需审批）
  │
  ├─ 风险等级 = medium
  │   └─→ 🟡 单人审批
  │         ├─ 审批通过 → 执行
  │         └─ 审批拒绝 → 返回拒绝信息给用户
  │
  └─ 风险等级 = high
      └─→ 🔴 双人审批
            ├─ 两人都通过 → 执行
            ├─ 任一人拒绝 → 返回拒绝信息
            └─ 超时 → 自动拒绝
```

### 8.3 Python Agent 引擎多实例部署

因为 Agent 执行是有状态的长连接（gRPC 双向流），多实例部署时需要：

1. **会话亲和性**: 同一个 session_id 的请求路由到同一个 Python 实例
2. **无状态化**: 会话上下文存在 Redis/PostgreSQL，实例重启不丢
3. **弹性伸缩**: 根据活跃会话数自动扩缩容
