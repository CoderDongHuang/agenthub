## 1. 开发总览

### 原则

```
🚫 不做：一步到位做完美 → 永远做不完
✅ 要做：每个 Phase 产出可演示的东西 → 快速验证 → 迭代
```

### 四阶段

```
Phase 0 (1周)  → 搭架子，三端能通信
Phase 1 (4周)  → 核心闭环：创建 Agent → 对话执行
Phase 2 (3周)  → 审批卡点 + 审计日志（核心差异化）
Phase 3 (3周)  → 工具市场 + 多渠道接入
Phase 4 (2周)  → 打磨、文档、开放 GitHub
```

## 4. Phase 2: 企业级功能

> **目标**: 实现审批卡点 + 审计日志 + 多租户隔离 — 这是和所有竞品的本质区别

### 第 1 周: 工具风险管控 + 审批流引擎

#### 后端（Java）

1. **工具注册与管理**
   - `ToolDefinition.java` 实体
   - `POST /api/tools` — 注册工具（开发者）
   - `GET /api/tools` — 工具列表（市场）
   - 工具风险等级（low / medium / high）

2. **工具审批策略**
   - `ToolApprovalPolicy.java` 实体
   - 配置：哪个风险等级 → 需要几级审批 → 谁来审批
   - `POST /api/tools/policies` — 配置审批策略

3. **审批流引擎**（使用 Flowable）
   - `ApprovalRequest.java` 实体
   - `ApprovalNode.java` 实体
   - `POST /api/approvals` — 发起审批
   - `POST /api/approvals/{id}/approve` — 通过
   - `POST /api/approvals/{id}/reject` — 拒绝
   - `GET /api/approvals/pending` — 我的待审批列表

4. **gRPC — 审批卡点实时通信**
   - Python 引擎在执行工具前 → 通过 gRPC 向 Java 查询是否需要审批
   - 需要审批 → Java 创建审批工单 → Python 暂停执行
   - 审批通过 → Java 通过 gRPC 通知 Python → Python 继续执行

## 附录 B: 关键风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| Python 引擎执行 Agent 时崩溃 | 中 | 高 | gRPC 健康检查 + 自动重启 + Java 端降级提示 |
| LLM API 不稳定/超时 | 高 | 中 | 重试机制 + 降级模型 + 超时后友好提示 |
| gRPC 双向流复杂度超过预期 | 中 | 高 | 备选方案：HTTP SSE 单向流 + HTTP 回调做审批通知 |
| 审批流引擎 (Flowable) 学习成本高 | 中 | 中 | 先用简单的状态机实现，后续再迁移到 Flowable |
| 个人开发时间不足 | 高 | 高 | Phase 1 完成就发布 MVP，不追求完美再发布 |
